Thread Safe Issue for Python strptime function

背景

实习的某一天，自己负责的报表服务需要进行分库分表来应对日渐膨胀的数据量，经过讨论， 决定先采用按周分表的策略，也就是数据入库的时候会根据对应日期映射到不同的周， 再根据不同的周数找到或创建不同的 table 。

根据不同日期计算对应的周是一个很简单的问题： 在该服务的设计中，每行数据对应的日期是由一个八位字符串来表示的， 如 20180811 表示2018年8月11日。 所以我们需要调用 strptime 先将字符串转化为 datetime 对象， 再调用 Python 中的一些日期库计算这是一年中的第几周。将这一周作为后缀名， 将数据插入对应的分表中。如，表 table1 中的一行 r1 中的日期为 20180811， 对应 2018 的第 32 周，那么行 r1 应该被插入表 table1_201832 中。

问题

实际操作中，有多张表需要同时插入数据，因此会创建多个线程，每个线程负责插入数据到一张表中。 每个线程在计算的时候会针对要入库的数据计算要插入的表名。实践中发现采用分表策略的服务在进行数据插入操作时， 有一定概率出现线程抛出异常的问题， 经过排查后发现，如果 strptime 会处于被多个线程同时调用的状态，会抛出异常。

Python 的 bug 列表中也报告了这个问题

Issue7980: time.strptime not thread safe

解决方案

目前，这个 issue 仍然处于 Open 状态，还未被最新一版的 Python 修复。但是已经有了合适的解决方案：

Lib/_strptime.py itself should be thread-safe. I am guessing that it has something to do with the way the C code in time.c imports _strptime.py. A possible solution if it is the C code’s import stuff is to create a time.py that imports a _time.c, but that’s a total guess as to whether that will solve anything. From https://bugs.python.org/issue7980

简而言之，这一 bug 的产生和 Python 调用的 C 函数是非线程安全的有关，解决方案是： 在导入 datetime, time 这些类库前，导入 _strptime。
